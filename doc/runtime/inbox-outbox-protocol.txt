Протокол Inbox–Outbox


Внешний выход

`outbox.zip`

Содержимое:

- `outbox.json`  
- provided files  


Внутренний output

- `trace.json`  
- `events.jsonl`  
- `plan.json` (A13)  


A13 ДОПОЛНЕНИЕ

Каждый run создаёт `plan.json`:

- сохраняется в runtime artifacts  
- связан с history через `planRelPath`  


A14 ДОПОЛНЕНИЕ

Если compiled plan структурно некорректен:

- execution не запускается  
- outbox формируется как ошибка  

Тип ошибки:

`PLAN_SCHEMA_INVALID`


A15 ДОПОЛНЕНИЕ

Начиная с A15:

- внутренний результат выполнения нормализуется до сборки внешнего ответа  
- `outbox.json` получает данные из общего слоя результата  
- внешний контракт `outbox.json` сохранён совместимым  

Это означает:

- `outbox.json` не должен сам вычислять итог выполнения  
- `outbox.json` является внешней поверхностью результата, а не источником истины  


Outbox contract

`outbox.zip` всегда содержит:

- `outbox.json`  
- provided files (если есть)  


Код завершения

`exitCode`:

- `0` → success  
- `1` → error  

`exitCode` отражает итог завершения запуска.

`error.code` отражает тип ошибки.

Эти значения относятся к разным уровням контракта и не должны смешиваться.


Получатель (receiver)

`receiver: uri`


Правило

URI обрабатывает только пакеты, у которых `receiver == uri`.
