Materialize Plan

RUNBOOK преобразуется в plan:

- нормализация  
- структурная подготовка  
- сохранение `plan.json`  

Каждый run создаёт plan до выполнения.


Plan Validation (A14)

На этом этапе runtime проверяет структурный контракт plan:

- plan является объектом  
- присутствует `plan.version`  
- `plan.steps` является массивом  
- каждый шаг (`step`) имеет корректную минимальную структуру  

Если проверка не пройдена:

- execution не начинается  
- возвращается ошибка `PLAN_SCHEMA_INVALID`  


Execution

Выполнение происходит через `run-plan`.

Runtime не исполняет runbook напрямую.


Normalize Result (A15)

После выполнения runtime строит единый внутренний результат запуска.

На этом этапе нормализуются:

- `ok`  
- `exitCode`  
- `error`  
- `attempts`  
- прочие поля итогового результата  

Задача этапа:

- убрать дублирование логики между runtime, CLI и history  
- получить единую внутреннюю форму результата  


Finalize Run (A15)

После нормализации результата runtime выполняет финализацию запуска.

На этом этапе:

- подготавливаются terminal surface и CLI surface  
- строится итог для history  
- подготавливается внешний outbox-отчёт  

Важно:

- terminal surface не должен заново вычислять итог выполнения  
- history не должен заново вычислять итог выполнения  
- outbox не должен становиться отдельным источником истины  


History

History сохраняет:

- trace  
- outbox  
- `planRelPath`  

`planRelPath` указывает на `plan.json`.
