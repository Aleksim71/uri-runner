Детали фаз


Intake Gate

URI отслеживает папку загрузки (download / intake folder)
и обнаруживает входящие пакеты (например inbox.zip).

Перед переносом пакета в Inbox URI выполняет intake-проверку:

1. Открывает пакет без полного распаковывания
2. Извлекает RUNBOOK.yaml (или metadata YAML)
3. Читает поле receiver

Правила:

- если receiver == uri → пакет принимается и переносится в Inbox
- если receiver != uri → пакет игнорируется
- если metadata отсутствует или некорректна → пакет отклоняется

Важно:

Пакеты, не адресованные URI, не попадают в Inbox и не запускают выполнение.

Сценарий выполнения начинается только после успешного прохождения intake-проверки.


Inbox Intake

URI принимает пакет, прошедший intake gate, и размещает его в Inbox.


Scenario Decode

RUNBOOK.yaml извлекается и валидируется.


Plan Compile

RUNBOOK компилируется во внутренний execution plan.


Run Sandbox Initialization

Для run создаётся отдельная runtime-песочница.

Структура:

runtime/runs/<runId>/

Создаваемые подкаталоги:

traces
artifacts
provided
logs
tmp

Назначение:

изолировать execution-артефакты
предотвратить загрязнение между run
обеспечить детерминированное хранение trace-данных


Baseline Capture

Фиксируется начальное состояние системы.


Environment Reset

Необязательная фаза подготовки окружения, выполняемая
перед первым шагом сценария.

Срабатывает, когда:

runtime.environment.reset_before_run = true

Pipeline сброса:

1 stop managed processes
2 cleanup runtime state
3 start managed server
4 run healthcheck

Назначение:

обеспечить детерминированное окружение выполнения
предотвратить загрязнение состояния между запусками


Execution

Выполняются шаги, определённые в секции execute RUNBOOK.


Restore / Rollback

При необходимости состояние системы восстанавливается.


Baseline Verification

Проверки подтверждают стабильность окружения.


Outbox Packaging

URI создаёт пакет результата выполнения.

outbox.zip

Содержимое:

outbox.json
опциональные provided files


History Persistence

Execution-артефакты и метаданные сохраняются в runtime history.

Артефакты, записываемые на этой фазе:

trace.json
execution-trace, сформированный из execution events

outbox.zip
пакет результата выполнения

history index

runtime/history/index.json

Индекс истории обеспечивает быстрый доступ к завершённым run
и поддерживает CLI-команды:

uri history
uri last
uri show <runId>


Trace-файлы остаются каноническим источником истины для каждого run.

History index — это компактный реестр, указывающий на trace-артефакты.
