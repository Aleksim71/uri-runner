URI Runner
Архитектура Runtime

Версия: 6 (A13 + A14 + A15 + A16 File Delivery Report V1)

Обзор

Архитектура Runtime описывает ядро выполнения URI Runner.

URI — движок выполнения, основанный на артефактах.

Ключевые изменения

A13:
Выполнение основано на materialized plan.
Runbook больше не исполняется напрямую.

A14:
Добавлена проверка структуры compiled plan перед выполнением.

A15:
Добавлен единый внутренний контракт результата выполнения.
Результат нормализуется до построения внешних поверхностей.

A16 / File Delivery Report V1:
Реализован отдельный handoff-слой классификации обязательных файлов.
Он не меняет execution-ядро, а усиливает surface path после выполнения.

Pipeline выполнения

RUNBOOK
→ compile
→ materialize plan
→ plan validation (A14)
→ run-plan
→ execution
→ normalize result / finalize runtime result (A15)
→ collect / classify required files (V1)
→ trace / history / outbox

Где находится новый слой

Слой Required Files Delivery Report V1:
- не относится к Intake Gate watcher-уровня
- не относится к compile
- не относится к plan validation
- не заменяет execution
- относится к handoff / surface path между execution result и финальной сборкой outbox

Это важно, потому что в A15 уже зафиксирован единый внутренний result layer.
Новый слой должен усиливать его, а не строить параллельную систему.

Проверка плана (A14)

Перед запуском execution runtime проверяет структурную целостность plan:

- plan должен быть объектом
- `plan.version` обязателен
- `plan.steps` должен быть массивом
- каждый step должен иметь корректную минимальную структуру

Если проверка не пройдена:

- execution не начинается
- run завершается с ошибкой `PLAN_SCHEMA_INVALID`

Важно

Проверка плана проверяет только структуру.

Она не заменяет:
- проверку существования команд
- проверку root policy
- runtime-ограничения (например `max_steps`)
- проверку полноты handoff-файлов для Макса

Единый контракт результата (A15)

Начиная с A15, runtime опирается на единый внутренний контракт результата выполнения.

Этот внутренний результат используется как источник истины для:
- terminal surface
- history persistence
- внешнего outbox-отчёта

Разделение ответственности:
- internal result — внутренняя нормализованная форма результата
- terminal output — отображение результата для CLI/терминала
- persisted artifacts — сохраняемые артефакты выполнения
- handoff delivery report — внешний отчёт по requested files для Макса

Важно:
- `exitCode` — это код завершения запуска
- `error.code` — это тип ошибки
- `fileDeliveryReport.ok` — это достижение цели по обязательному набору файлов
- эти значения не тождественны и не должны смешиваться

A15 не вводит параллельную runtime-систему.
V1 тоже не вводит параллельную runtime-систему.
Оба слоя усиливают существующий pipeline.

Required Files Delivery Report V1

Назначение слоя:
- собрать обязательные requested files
- по каждому файлу выдать один из статусов: `provided`, `missing`, `failed`
- собрать summary
- при `ok=false` обязательно приложить дерево проекта
- сохранить полезные уже найденные файлы

Решение о цели:
- все requested files = `provided` → `ok=true`
- хотя бы один `missing` или `failed` → `ok=false`

Система артефактов

Основные execution-артефакты:

PLAN (`plan.json`)
TRACE
RESULT
META
OUTBOX

Для handoff-доставки обязательных файлов дополнительно допустимы:
- `provided/*`
- `provided/project-tree.txt`
- `outbox.json.fileDeliveryReport`

Гарантии

- каждый run создаёт plan artifact до выполнения
- execution всегда идёт через plan
- history содержит `planRelPath`
- итог выполнения нормализуется до записи во внешние поверхности
- handoff delivery layer не удаляет уже найденные полезные файлы
- при `fileDeliveryReport.ok=false` дерево проекта обязательно прикладывается

Взаимодействие с AI

URI взаимодействует с AI через протокол Inbox–Outbox.

Для V1 это означает:
- Макс получает один главный внешний отчёт — `outbox.json`
- к нему могут быть приложены найденные файлы и дерево проекта
- диагностическая полезность не отменяет бинарную оценку цели
